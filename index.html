整理时间：2026-06-15；内容：Docker 挂载、代理、CA 证书与 Docker daemon 网络排查。
用户 需求

目前我通过复杂的网络连接办法，在公司内部机器连接到了内部服务器。我现在想 vllm-ascend 中的 docker：

# Update DEVICE according to your device (/dev/davinci[0-7])

docker run --rm \

# Install curl
apt-get update -y && apt-get install -y curl

并挂载到 ~/lhy/vllm-ascend0615，怎么修改命令？

Codex 回答

在 docker run 里加一个 -v 挂载即可。建议这样写，把宿主机的 ~/lhy/vllm-ascend0615 挂到容器里的同一路径，并进入该目录：

mkdir -p ~/lhy/vllm-ascend0615

export DEVICE=/dev/davinci0
export IMAGE=quay.io/ascend/vllm-ascend:v0.20.2rc1

docker run --rm \
  --name vllm-ascend \
  --shm-size=1g \
  --device $DEVICE \
  --device /dev/davinci_manager \
  --device /dev/devmm_svm \
  --device /dev/hisi_hdc \
  -v /usr/local/dcmi:/usr/local/dcmi \
  -v /usr/local/bin/npu-smi:/usr/local/bin/npu-smi \
  -v /usr/local/Ascend/driver/lib64/:/usr/local/Ascend/driver/lib64/ \
  -v /usr/local/Ascend/driver/version.info:/usr/local/Ascend/driver/version.info \
  -v /etc/ascend_install.info:/etc/ascend_install.info \
  -v /root/.cache:/root/.cache \
  -v ~/lhy/vllm-ascend0615:/root/lhy/vllm-ascend0615 \
  -w /root/lhy/vllm-ascend0615 \
  -p 8000:8000 \
  -it $IMAGE bash

进入容器后你就在 /root/lhy/vllm-ascend0615，这个目录和宿主机的 ~/lhy/vllm-ascend0615 是同步的。

用户 报错
docker: Error response from daemon: failed to resolve reference "quay.io/ascend/vllm-ascend:v0.20.2rc1": failed to do request : Head "https://xxx": proxy connect tcp: dial tcp 120.63.69.62:3130: connect : no route to host

但是我现在这个环境 curl 能得到，curl quay.io/v2/ascend/vvlm-ascend/manifests/v0.20.2rc1 返回 301（我觉得是正常）。

Codex 排查方向

这个报错看起来不是镜像名本身的问题，而是 Docker daemon 正在走一个到不了的代理；当前 shell 里的 curl 能通，并不代表 Docker daemon 的网络也能通。还有一个小细节：curl 里的 vvlm-ascend 似乎多了一个 v，镜像名应是 vllm-ascend。

检查 Docker daemon 当前代理：

docker info | grep -i proxy
systemctl show --property=Environment docker

再查代理配置来源：

sudo grep -Rni proxy /etc/systemd/system/docker.service.d /etc/docker/daemon.json /root/.docker/config.json ~/.docker/config.json 2>/dev/null

如果里面有旧的 120.63.69.62:3130，要么改成当前服务器能访问的代理，要么去掉它。验证 quay 地址建议使用：

curl -I -L https://quay.io/v2/ascend/vllm-ascend/manifests/v0.20.2rc1
用户 补充

代理应该是正常的，http://账号:密码@120.63.69.62:3130 应该没问题（我 baidu curl 成功了，不然百度都 curl 不了）。但是：

curl -I -L https://quay.io/v2/ascend/vllm-ascend/manifests/v0.20.2rc1

返回：

http/1.1 200 connection established
proxy-agent:netentsec
connection:close
curl:(77) error setting certificate file:/ets/ssl/certs/ca-cerificates.crt
Codex CA 证书判断

这个新报错里，最关键的不是 proxy，而是：

curl: (77) error setting certificate file: /ets/ssl/certs/ca-cerificates.crt

这是本机 curl 的 CA 证书路径配置错了。正常路径一般是：

/etc/ssl/certs/ca-certificates.crt

可以先看是谁设置错了：

env | grep -Ei 'SSL_CERT|CURL_CA|REQUESTS_CA|proxy'

如果看到类似 CURL_CA_BUNDLE 或 SSL_CERT_FILE 指向错误路径，先临时取消：

unset CURL_CA_BUNDLE
unset SSL_CERT_FILE

然后显式通过代理测试：

curl -I -L \
  -x 'http://账号:密码@120.63.69.62:3130' \
  https://quay.io/v2/ascend/vllm-ascend/manifests/v0.20.2rc1
用户 澄清

c 是我自己打错了，他的路径正确。忽略笔误。

Codex 继续定位

明白，那就先忽略拼写。这里其实有两条线：

第一，curl 已经连上代理了。这一句说明代理 CONNECT 成功：

HTTP/1.1 200 Connection established
proxy-agent: netentsec

所以 120.63.69.62:3130 从当前 shell 看，大概率是通的。

第二，curl: (77) 是本机 CA 文件加载问题。如果路径确实正确，建议检查文件本身：

ls -l /etc/ssl/certs/ca-certificates.crt
file /etc/ssl/certs/ca-certificates.crt
head -n 2 /etc/ssl/certs/ca-certificates.crt

可以修一下 CA 包：

sudo apt-get update
sudo apt-get install --reinstall -y ca-certificates
sudo update-ca-certificates

但 Docker 的问题要单独看。原始 Docker 报错是：

dial tcp 120.63.69.62:3130: connect: no route to host

这发生在 Docker daemon 连接代理阶段，甚至还没走到证书验证。也就是说：当前终端 curl 能连代理，不代表 dockerd 也能连。

建议直接检查 Docker daemon 吃到的代理配置：

docker info | grep -i proxy
systemctl show docker --property=Environment

如果账号密码里有特殊字符，尤其是 @、:、#、%、/、?，Docker 的 systemd 配置里必须 URL 编码。

判断重点：普通 curl、sudo curl、Docker daemon 三者可能处在不同网络/环境变量上下文里，要分别验证。
